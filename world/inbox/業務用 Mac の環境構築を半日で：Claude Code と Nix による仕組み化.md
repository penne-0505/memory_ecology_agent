## はじめに

業務用の Mac を新しくしたので、先日セットアップした。

これまでは移行アシスタントを使うことが多かったが、せっかくの新端末なので身軽に使えるようゼロから行った。  
とはいえ、インストールするアプリの選別、ライセンスの移行、dotfiles の整理、開発環境の再構築などやることは多い。

今回の端末変更では、 **アプリ等の移行および開発環境構築を約半日で完了** できた。

この記事では、その鍵となる以下の2つの仕組みと実際の流れ、そして残った改善点をまとめる。

1. Claude Code のカスタムスキル（ `iko-assist` ）による **事前計画の自動生成**
2. チームで整備している Nix ベースのツールによる **開発環境構築のコード化**

## 移行アシスタントを使わない理由

macOS には公式の移行アシスタントがある。旧端末のデータをまるごと新端末に転送できて便利だが、以下の理由から今回は使用しなかった。

- 旧環境の汚れを持ち込まないため
	- 長年使ったMacには使っていないアプリや古い設定ファイルが残存。移行アシスタントはそれらも引き継いでしまう
- データ移行の時間的コスト
	- 移行アシスタントは数百GB~数TBのデータをまるっと引き継ぐので有線接続でもそこそこ時間がかかる
		- 移行の間は両端末が占有され、電源への接続が不可欠であるなど、時間・空間的な準備も必要

## 事前計画の立案（iko-assist スキル）

ゼロから手作業でセットアップと決めたはよいが、段取りを考えるのも手間なので、 Claude にスキルを定義し、事前計画を自動化した。  
余談だが、スキル名は本家である移行アシスタントの本歌取りである。

## iko-assist スキル

`iko-assist` は自作した Claude Code のカスタムスキル（ [全体のファイルはこちら](https://github.com/gaussbeam/agent-skills/tree/main/claude_samples/skills/iko-assist) ）。  
旧端末の環境情報（ `brew list` の出力、言語ランタイムのバージョン、システム設定の実測値など）をインプットとして受け取り、新端末のセットアップに必要なファイルを自動生成する。

| ファイル | 内容 |
| --- | --- |
| セットアップ手順.md | 全体の作業手順とそのチェックリスト |
| アプリリスト.md | インストール要否の振り分け（する/しない/要判断） |
| Brewfile | `brew bundle` で一括インストールする定義ファイル |
| 開発環境.md | 言語ランタイム・dotfiles・APIトークンの設定手順 |
| システム設定.md | `defaults write` コマンド群 + GUI確認項目 |

## AI による計画立案のメリット

自分でアプリや設定を整理する場合、「どのアプリを使っている？」「 Mac のどの設定をいじったか？」など確認することが多い。

`iko-assist` は旧端末のインストール済みアプリを列挙し、用途・入手方法とともに要否を整理する。  
また、 Finder や Mission Control などの各種設定は、現在の値を確認したうえで、可能な限りコマンド化を行う。（ Dock・Finder・トラックパッド設定はすべてコマンド化可能）  
Night Shift や FileVault など、一部は GUI での設定が必要だが、それらもチェックボックス付きの TODO リストとして書き出される。

**旧端末の設定を確認する手間がなくなり、かつ次に何をすべきか迷わない状態を作れた** のは大きなメリットだった。

### システム設定の自動適用例

```
defaults write com.apple.dock autohide -bool true     # Dockを自動で非表示
defaults write com.apple.dock wvous-tl-corner -int 0  # ホットコーナー無効
defaults write com.apple.finder CreateDesktop false   # デスクトップアイコン非表示
killall Dock && killall Finder
```

## 開発環境の構築（Nix）

`iko-assist` により、 Mac のセットアップはほぼ自動化 or TODOリスト化できる。  
とはいえ業務端末としては開発環境の構築が欠かせない。弊社にはNixを用いて環境構築を自動化する仕組みがあるので、ありがたく使わせてもらった。

## 環境構築を自動化する仕組みの概略

Teachme Biz の開発チームでは、 `nix run` で実行できるセットアップ手順を用意している。

nix-darwin と home-manager を組み合わせ、Homebrew パッケージ・macOS のシステム設定・ユーザー環境設定を Nix の設定ファイル群として宣言的に管理している。  
1 コマンドでパッケージのインストールからコンテナのセットアップまで一括で行える。  
また、設定変更後も同じコマンドを再実行するだけで冪等に適用される。

## 課題と改善点

## iko-assist と Nix の重複

`iko-assist` の生成物には、rbenv・bun などのインストール手順が含まれる。一方、Nix の設定でも同様のツールを管理できる。現状では両者が独立しており、二重に設定作業が発生する。

プライベートの端末であれば `iko-assist` だけで事足りるが、業務用端末でセットアップ用のスクリプト等が定義済みの場合は、処理の重複や設定の衝突がないかを考慮する必要がある。

## Mac 本体設定の引き継ぎ粒度

現在の `iko-assist` は「私がどの設定をカスタマイズしているか」を旧端末から読み取り、同じ設定を新端末に再現しようとする。  
macOS は十人十色のカスタマイズがされるので、現在の SKILL は別の人にとっては冗長であったり、私が変更していない設定項目の見落とす可能性を否定できない。

よく変更される設定項目のみを厳選してヒアリングし、それ以外はユーザ自身の手に委ねるのも手かもしれない。

## 〆

今回の端末変更を半日で終えられた要因は以下の3点である。

- 移行アシスタントを使わず、ゼロから再構築することで無駄な待機時間を排除した
- `iko-assist` によって「何をどの順番でやるか」の思考コストをゼロにした
- Nix による開発環境のコード化で、プロジェクト固有の再現が自動化されていた

また、移行アシスタントを使わなかったことで、新端末のセットアップ中も旧端末は自由になったので、ちょこちょこ作業を進められた点も良かった。  
改善点はあるものの、「端末のセットアップはしんどい作業」という認識が変わる体験だった。スキルとして整備しておくと、次の移行時はさらにスムーズにできるだろう。
